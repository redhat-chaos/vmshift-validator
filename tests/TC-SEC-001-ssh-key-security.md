# TC-SEC-001: SSH Key Security

## Test ID
TC-SEC-001

## Test Name
SSH Key Handling and Security Controls

## Feature
Security — SSH key file permissions, cloud-init injection, log exposure prevention, and SSH client options.

## Objective
Verify that SSH private keys have restrictive file permissions, public keys are correctly injected into VM cloud-init without leaking into logs, private key content is never exposed in script output, and SSH client options (`StrictHostKeyChecking`, `UserKnownHostsFile`) are configured with awareness of the security tradeoffs.

## Preconditions
1. SSH key pair has been generated via `make generate-keys` or manually placed in `keys/`.
2. The private key file is `keys/kube-burner` and public key is `keys/kube-burner.pub`.
3. Source cluster is available for cloud-init verification scenarios.
4. Scripts in `scripts/` and `scripts/lib/` are present and unmodified.

## Test Data
| Data Item | Value |
|-----------|-------|
| Private key path | `keys/kube-burner` |
| Public key path | `keys/kube-burner.pub` |
| Key type | ed25519 (from `ssh-keygen -t ed25519`) |
| Expected private key permissions | `600` or `400` |
| SSH user | `fedora` |

## Steps

### Sub-case 1.1: Private Key File Permissions

#### Step 1: Generate keys via make target
```bash
rm -f keys/kube-burner keys/kube-burner.pub
make generate-keys
```

#### Step 2: Check file permissions
```bash
stat -c '%a' keys/kube-burner  # Linux
# or
stat -f '%A' keys/kube-burner  # macOS
# Expected: 600 (rw-------)
```

#### Step 3: Verify SSH refuses overly permissive keys
```bash
chmod 644 keys/kube-burner
virtctl ssh fedora@vm/test-vm -n vm-services -i keys/kube-burner --command "true" 2>&1
# SSH should warn: "Permissions 0644 for 'keys/kube-burner' are too open"
# and refuse to use the key
```

#### Step 4: Restore correct permissions
```bash
chmod 600 keys/kube-burner
```

---

### Sub-case 1.2: Public Key Injection via Cloud-Init

#### Step 1: Verify public key in rendered kube-burner config
```bash
make render-config
grep -c "$(cat keys/kube-burner.pub)" kube-burner/.rendered-vm-services.yml
# Expected: > 0 (public key appears in cloud-init ssh_authorized_keys)
```

#### Step 2: Verify inside a running VM
```bash
make density-setup
virtctl ssh fedora@vm/vm-svc-0 -n vm-services -i keys/kube-burner \
  --command "cat ~/.ssh/authorized_keys"
# Output should contain the public key string
```

#### Step 3: Verify key type matches
```bash
virtctl ssh fedora@vm/vm-svc-0 -n vm-services -i keys/kube-burner \
  --command "cat ~/.ssh/authorized_keys" | awk '{print $1}'
# Expected: ssh-ed25519 (matches key type generated by make generate-keys)
```

---

### Sub-case 1.3: SSH Key Not Logged or Exposed in Output

#### Step 1: Run density-setup and capture all output
```bash
make density-setup 2>&1 | tee /tmp/density-output.log
```

#### Step 2: Search for private key content in output
```bash
grep -c "PRIVATE KEY" /tmp/density-output.log
# Expected: 0 (private key header never appears in output)

grep -c "$(head -2 keys/kube-burner | tail -1)" /tmp/density-output.log
# Expected: 0 (private key data never appears)
```

#### Step 3: Run migration and capture output
```bash
make migrate-selective VMS=vm-svc-0 2>&1 | tee /tmp/migrate-output.log
```

#### Step 4: Search migration output for key material
```bash
grep -c "PRIVATE KEY" /tmp/migrate-output.log
# Expected: 0

# Public key may appear in verbose/debug logs via the render-config step,
# but should NOT appear at LOG_LEVEL=1 (info)
grep -c "$(cat keys/kube-burner.pub)" /tmp/migrate-output.log
# Expected: 0 at LOG_LEVEL=1
```

#### Step 5: Test with debug logging
```bash
make density-setup LOG_LEVEL=3 2>&1 | tee /tmp/debug-output.log
grep -c "PRIVATE KEY" /tmp/debug-output.log
# Expected: 0 (even at debug level, private key must not be exposed)
```

---

### Sub-case 1.4: SSH Client Options — StrictHostKeyChecking

#### Step 1: Examine ssh.sh library defaults
```bash
grep -n "StrictHostKeyChecking" scripts/lib/ssh.sh
# Should find: -o StrictHostKeyChecking=no or StrictHostKeyChecking=accept-new
```

#### Step 2: Examine migrate-parallel.sh defaults
```bash
grep -n "StrictHostKeyChecking" scripts/migrate-parallel.sh
# LOCAL_SSH_OPTS="-o StrictHostKeyChecking=accept-new"
```

#### Step 3: Verify UserKnownHostsFile=/dev/null is set
```bash
grep -n "UserKnownHostsFile" scripts/lib/ssh.sh
# Should find: -o UserKnownHostsFile=/dev/null
```

#### Step 4: Document the security tradeoff
- StrictHostKeyChecking=no/accept-new: Accepts any host key without verification.
  - **Risk**: Vulnerable to man-in-the-middle attacks.
  - **Justification**: VMs are ephemeral with new host keys on each boot; known_hosts would accumulate stale entries and cause SSH failures.
- UserKnownHostsFile=/dev/null: Discards learned host keys.
  - **Risk**: Cannot detect host key changes (potential MITM indicator).
  - **Justification**: Test VMs are recreated frequently; persistent host key tracking provides no security value in this context.

---

### Sub-case 1.5: Key Path Passed Securely to Scripts

#### Step 1: Verify scripts receive key path via argument, not environment
```bash
grep -n "ssh-key\|SSH_KEY\|identity-file" scripts/density-setup.sh scripts/migrate-single-vm.sh scripts/lib/ssh.sh
# Scripts receive --ssh-key as CLI argument, stored in local variable
# Not exported to child processes beyond virtctl invocation
```

#### Step 2: Verify virtctl invocation uses --identity-file
```bash
grep -n "identity-file" scripts/lib/ssh.sh scripts/lib/executor.sh
# Should use: --identity-file="$SSH_KEY"
# Key path is properly quoted to prevent word splitting
```

## Expected Result
| Sub-case | Expected Behavior |
|----------|-------------------|
| 1.1 — Permissions | Private key has mode 600; SSH rejects if permissions are too open |
| 1.2 — Cloud-init injection | Public key appears in VM's authorized_keys |
| 1.3 — No log exposure | Private key content never appears in any output at any log level |
| 1.4 — SSH options | StrictHostKeyChecking=no/accept-new and UserKnownHostsFile=/dev/null are set |
| 1.5 — Secure passing | Key path passed via --identity-file, properly quoted |

## Validation Points
- [ ] `make generate-keys` creates private key with mode 600 (not 644 or 755).
- [ ] `ssh-keygen` is invoked with `-N ""` (empty passphrase) — documented security choice.
- [ ] The public key string in rendered kube-burner config matches `keys/kube-burner.pub`.
- [ ] No output from any script at LOG_LEVEL=1 contains the private key path value.
- [ ] No output at any LOG_LEVEL contains actual private key data (base64-encoded key material).
- [ ] The `--identity-file` flag is used (not `-i` shorthand which could be confused with other flags).
- [ ] `LOCAL_SSH_OPTS` defaults include both `StrictHostKeyChecking` and `UserKnownHostsFile` settings.
- [ ] Key path variables are double-quoted in all script usages to prevent globbing.
- [ ] The `.gitignore` includes `keys/` directory to prevent accidental key commits.
- [ ] `scripts/lib/ssh.sh` `run_on_vm()` function sends diagnostics to stderr, not stdout (no key path leakage in captured output).

## Acceptance Criteria
1. Private key files are created with restrictive permissions that SSH accepts.
2. Private key content (headers and material) never appears in stdout, stderr, or log files at any verbosity level.
3. Public key injection via cloud-init is verified by successful SSH authentication.
4. SSH options are explicitly set (not relying on system defaults).
5. The `keys/` directory is gitignored to prevent accidental repository exposure.

## Edge Cases Covered
- Key file with spaces in path: `SSH_KEY="keys/my kube-burner key"` — quoting must handle it.
- Key regeneration: Running `make generate-keys` when key already exists (refuses to overwrite).
- Key permissions on NFS mount: Some network filesystems don't honor chmod — SSH may still reject.
- Empty public key (SSH_PUBLIC_KEY is empty): `render-config` catches this and errors.
- Windows-style line endings in key file: SSH may reject keys with `\r\n`.

## Failure Scenarios
| Failure | Impact | Detection |
|---------|--------|-----------|
| Key logged at debug level | Private key leaked to terminal/CI logs | grep for "PRIVATE KEY" in captured output |
| chmod not applied | SSH refuses key; all VM access fails | stat shows wrong permissions |
| Public key not injected | SSH auth fails; workload stabilization times out | VM authorized_keys is empty |
| StrictHostKeyChecking omitted | SSH prompts for confirmation, hanging automation | Script hangs on first SSH to new VM |
| Key path unquoted | Word splitting breaks path with spaces | SSH error: no such file |

## Automation Potential
**High**. Most checks are scriptable:
- Permission checks: `stat` + assertion.
- Log exposure: `grep -c "PRIVATE KEY"` on captured output.
- Cloud-init injection: Parse rendered YAML with yq.
- SSH option verification: grep script source files.
- No cluster needed for permission and log-exposure checks.
- Estimated effort: 2–3 hours.

## Priority
**P1 — High**

## Severity
**S2 — Major**

SSH key mishandling could expose private keys in CI logs or to unauthorized users. While the keys are test-environment scoped, proper handling is essential for security hygiene and to prevent habits that could transfer to production systems.
